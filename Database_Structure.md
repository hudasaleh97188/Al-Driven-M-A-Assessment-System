# Database Structure & Relationships

The application uses a lightweight **SQLite** database (`deal_lens.db`) designed with a write-once/read-many architecture. Instead of highly normalizing the complex hierarchical data produced by the AI (which would require 15+ complicated tables), the database uses **3 simple tables** and relies on **JSON blobs** to store the rich, nested assessment data.

## Tables and Relationships

### 1. `users` Table
Stores users and their roles for RBAC.
| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | Unique identifier. |
| `username` | TEXT | UNIQUE, NOT NULL | Login name. |
| `password_hash` | TEXT | NOT NULL | Hashed password. |
| `role` | TEXT | CHECK(viewer, reviewer, admin) | User's role. |
| `created_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | |

### 2. `companies` Table
This is the master record for every target company analyzed in the system.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | Unique identifier for the company. |
| `name` | TEXT | NOT NULL, UNIQUE (NOCASE) | The name of the company. |
| `industry` | TEXT | Optional | Industry sector. |
| `created_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | Timestamp when the company was added. |
| `updated_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | Timestamp of the last update. |

### 3. `analysis_runs` Table
Stores the historical and active M&A assessment runs for a company. 
**Relationship:** `company_id` is a FOREIGN KEY referencing `companies(id)` with `ON DELETE CASCADE`. One Company -> Many Analysis Runs.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | Unique identifier for the run. |
| `company_id` | INTEGER | NOT NULL, FK to `companies(id)` | Which company this analysis belongs to. |
| `run_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | Timestamp of the analysis. |
| `status` | TEXT | CHECK IN ('pending','running','completed','failed') | Current state of the background processing. |
| `currency` | TEXT | Optional | Standardized currency (e.g., USD, EUR). |
| `result_json` | TEXT | Optional | Qualitative and unstructured AI output payload stored as JSON. |
| `error_message` | TEXT | Optional | Error details if the run failed. |

### 4. `financial_metrics` Table
Extracts the purely financial data out of the JSON blob into a structured normalized table.
**Relationship:** `analysis_run_id` is a FOREIGN KEY referencing `analysis_runs(id)`.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | Unique identifier. |
| `analysis_run_id` | INTEGER | NOT NULL, FK to `analysis_runs(id)`| The run this metric belongs to. |
| `category` | TEXT | Optional | e.g. 'Balance Sheet', 'Income Statement' |
| `metric_name` | TEXT | NOT NULL | e.g. 'Total Assets' |
| `is_calculated` | BOOLEAN | DEFAULT 0 | True for ratios/margins, false for foundational items. |
| `reported_currency` | TEXT | | Original currency. |
| `value_reported` | REAL | | Value in reported currency. |
| `value_usd` | REAL | | Value normalized to USD. |
| `year` | INTEGER | | e.g. 2023 |
| `confidence_score`| REAL | | AI confidence score in the extraction. |
| `explanation` | TEXT | | AI explanation. |

### 5. `financial_metrics_approved` Table
Stores human-reviewed overrides of financial data.
**Relationship:** `metric_id` FK to `financial_metrics(id)`. `reviewed_by` FK to `users(id)`.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | Unique identifier. |
| `analysis_run_id` | INTEGER | NOT NULL, FK to `analysis_runs(id)`| For quick filtering. |
| `metric_id` | INTEGER | NOT NULL, FK to `financial_metrics(id)`| The linked AI metric. |
| `value_usd_approved`| REAL | | The user-approved USD value. |
| `value_reported_approved`| REAL| | The user-approved local currency value.|
| `reviewed_by` | INTEGER | FK to `users(id)` | Which user approved/edited it. |
| `reviewed_at` | DATETIME| DEFAULT CURRENT_TIMESTAMP | |

### 6. `peer_ratings` Table
Stores the results of comparative peer ratings generated by the LLM.
**Relationship:** `company_id` is a FOREIGN KEY referencing `companies(id)` with `ON DELETE CASCADE`. One Company -> Many Peer Ratings.

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | Unique identifier for the rating run. |
| `company_id` | INTEGER | NOT NULL, FK to `companies(id)` | Target company rating relation. |
| `peer_company_name`| TEXT | | Peer name analyzed. |
| `run_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | Timestamp of the rating. |
| `result_json` | TEXT | Optional | The detailed peer comparison scores stored as JSON. |
